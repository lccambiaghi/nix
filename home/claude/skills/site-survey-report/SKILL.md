# Site Survey Report Generation

Generate a professional site survey report from a review workbook, producing a Word document that justifies a predictive maintenance investment for an industrial customer.

## Inputs

| Input | Required | Description |
|-------|----------|-------------|
| Review workbook | **Required** | `.xlsx` file with customer Q&A, component inventory, field observations, and media files |
| Pump bearing research report | Optional | PDF generated by the equipment-research pipeline (typically `pump_bearing_research_report.pdf`). Contains confirmed bearing part numbers, grease data, and sensor placement recommendations. **Overrides or supplements workbook component data when present.** |

If the user does not provide a bearing research PDF path, check for one at the conventional location:
```
/Users/luca/git/equipment-research/output/<customer_slug>/pump_bearing_research_report.pdf
```
If neither is found, proceed without it.

## Output

A filled `report.docx` in the customer's workspace folder, plus markdown section drafts.

## Pipeline

### Step 1 — Setup workspace

```
docs/reports/<customer_slug>/
├── tmp_data/    # Section drafts (.md) + extracted JSON data
└── output/      # Final deliverables
```

---

### Step 2 — Extract workbook data

Read the workbook directly (use the xlsx skill / openpyxl / pandas as needed) and explore its sheets before writing anything. Workbook schemas vary between customers — do not assume fixed sheet names or column positions.

**Exploration approach:**
1. List all sheet names.
2. For each sheet, read the first ~20 rows to understand its structure (headers, layout, any merged cells or multi-row headers).
3. Identify which sheets contain: Q&A / discovery data, component/asset inventory, observations or field notes, sizing or project scope data.

Once the structure is understood, extract and write the following four JSON files to `tmp_data/`:

**`qa.json`** — Q&A rows from the discovery/survey sheet(s). For each row capture:
```json
[{ "section_tag": "...", "question": "...", "answer": "..." }]
```
Include only rows that have a non-empty answer. The `section_tag` should map to the report section it feeds (e.g. `impianto`, `obiettivi`, `macchinari`, `dimensionamento`, `roi`). Infer it from column headers or question text if not explicit.

**`components.json`** — Asset/component inventory. Include only items that are approved and have a quantity > 0. For each item capture all available nameplate fields (asset ID, description, manufacturer, model, process/line, power, speed, voltage, year, any other relevant specs found in the sheet).

**`observations.json`** — Any rows flagged for inclusion in the report (look for a flag column, a "report" checkbox, or similar). Capture the observation text and any associated asset or area.

**`sizing.json`** — Project scope figures. Extract at minimum:
```json
{ "sensor_count": <int>, "gateway_count": <int>, "installation_days": <int> }
```
These may be on a summary sheet, a pricing tab, or computed from the component list — use judgment.

If a field genuinely cannot be found, set it to `null` and add a `"_note"` key explaining where you looked.

**Pre-compute ROI figures** and save to `tmp_data/roi.json`. The formulas are in `references/section-guides/06_roi.md`. You need these before spawning section agents so §5 and §6 have the actual numbers.

---

### Step 2b — Parse bearing research report (if available)

If a `pump_bearing_research_report.pdf` was provided or found, read it and extract its contents into `tmp_data/bearing_research.json`.

The JSON must capture:

```json
{
  "source": "pump_bearing_research_report.pdf",
  "date": "<report date>",
  "sites": ["<site name 1>", "<site name 2>"],
  "assets": [
    {
      "asset_id": "<e.g. P101>",
      "site": "<site name>",
      "motor_manufacturer": "<e.g. Ercole Marelli>",
      "motor_model": "<full model string>",
      "motor_power_kw": <number or null>,
      "motor_year": <year or null>,
      "bearing_DE": "<bearing designation, e.g. 7328-B>",
      "bearing_NDE": "<bearing designation, e.g. NU326-C3>",
      "bearing_confirmed": true/false,
      "bearing_confirmation_source": "<drawing / rating plate / test certificate / assumed>",
      "grease_interval_DE_h": <number or null>,
      "grease_qty_DE_g": <number or null>,
      "grease_type_DE": "<e.g. Lithium G3>",
      "grease_interval_NDE_h": <number or null>,
      "grease_qty_NDE_g": <number or null>,
      "grease_type_NDE": "<e.g. Lithium G3>",
      "notes": "<any special notes e.g. NDE insulated>"
    }
  ],
  "pumps": [
    {
      "asset_id": "<e.g. P101>",
      "site": "<site name>",
      "pump_manufacturer": "<e.g. Rotos>",
      "pump_model": "<e.g. PDV 600-B>",
      "bearing_type": "plain / rolling element",
      "bearing_notes": "<e.g. Babbitt metal, spherical radial — no BPFO/BPFI analysis>",
      "sensor_recommended": true/false,
      "sensor_notes": "<rationale>"
    }
  ],
  "sensor_summary": [
    {
      "site": "<site name>",
      "machine_type": "<e.g. Ercole Marelli 2794 kW>",
      "units": <count>,
      "sensors_per_unit": <count>,
      "subtotal": <count>,
      "rationale": "<e.g. Frame 630, DE-NDE >100cm — 2 motor sensors>"
    }
  ],
  "total_sensors_from_research": <number>,
  "discrepancy_with_workbook": "<note if research sensor count differs from sizing.json>"
}
```

**Conflict resolution rules:**
- If `total_sensors_from_research` differs from `sizing.json` sensor_count, flag the discrepancy in the JSON and **use the research report figure** as authoritative for §3 and §4 — the workbook may have been estimated before bearing confirmation.
- Bearing part numbers and grease data from the research report are confirmed from primary sources (nameplates, engineering drawings, test certificates); treat them as ground truth.
- If an asset appears in `components.json` but not in the research report, keep the workbook data; do not invent bearing specs.

If no PDF is available, skip this step entirely and leave `tmp_data/bearing_research.json` absent. Agents must handle its absence gracefully.

### Step 3 — Write sections in parallel (sub-agents)

Spawn all section agents **in a single turn** so they run concurrently. Each agent writes its section(s) to `sections/` and completes independently. The main agent then continues with Step 4.

**Pass each sub-agent:**
- The path to `tmp_data/` (so it can read `qa.json`, `components.json`, `observations.json`, `sizing.json`, `roi.json`)
- Whether `bearing_research.json` exists in `tmp_data/` (pass the path if it does)
- Which section(s) to write and the output filename(s)
- The section guide path from `references/section-guides/`
- The example file paths from `references/examples/`
- The writing rules below
- The **file format convention** for its output (see per-agent notes)

#### Writing rules (pass to every agent)

- **Language**: Italian
- **Plain text only**: no markdown formatting — no `**bold**`, `# headers`, `---` rules. The docx template handles all styling. Write only the narrative body.
- **Factual**: base everything on data from the JSON files. Do not fabricate.
- **Placeholders**: when critical info is missing, insert `[DA COMPILARE: brief description]`
- **Self-contained**: each agent does not know what others are writing. The main agent handles consistency in Step 4.

#### File format conventions (critical for rendering)

The render script (`scripts/render_docx.py`) parses section files by splitting on UPPERCASE HEADER LINES. Sub-agents must write files in exactly this format.

**Agent A** (`01_impianto.md`) — single block of text, no dividers needed.

**Agent B** (`02_obiettivi.md`) — write §2.1 first (3 paragraphs: intro/date/participants, transition sentence, bullet list of 6 objectives), then §2.2 after a blank line. The script splits by paragraph count: first 3 paragraphs = §2.1, the rest = §2.2.

**Agent C** (`03_macchinari.md`) — one block per process, each starting with an UPPERCASE title followed by `—`:
```
NOME DEL PROCESSO — DESCRIZIONE BREVE

...process narrative...

SECONDO PROCESSO — DESCRIZIONE BREVE

...second process narrative...
```

#### Special notes for specific agents

**Agent C**: group components by Process from `components.json`. One block per unique process. Each block follows the four-part structure: process overview → shared criticalities → pilot scope → per-asset specs. If `bearing_research.json` exists:
- Use confirmed bearing part numbers (DE/NDE), grease intervals, grease quantities, and grease types for each asset instead of whatever the workbook may contain.
- For pump assets with plain/Babbitt bearings, mention that vibration monitoring focuses on housing velocity/displacement rather than envelope analysis (no BPFO/BPFI fault frequencies apply).
- Note bearing confirmation source (engineering drawing, rating plate, test certificate) to give confidence to the reader.
- Where bearing data is unavailable (e.g., unknown manufacturer motors), write a placeholder `[DA COMPILARE: rilevazione targhetta motori in loco]`.

**Agent D**: read `sizing.json` for sensor_count and gateway_count. Read `components.json` for the per-asset walkthrough in §4.1. If `bearing_research.json` exists:
- Use `total_sensors_from_research` as the authoritative sensor count instead of `sizing.json` sensor_count if they differ (note the discrepancy).
- In §4.1, use the per-asset sensor rationale from `bearing_research.json` (`sensor_summary[].rationale`) to explain sensor placement decisions (e.g., motor frame size justifying DE+NDE separation).
- Cross-reference bearing types when explaining sensor technology choices in §4.3 (e.g., plain bearings require different analysis than rolling element bearings).

**Agent E**: read `roi.json` for benefit ranges, ROI, and payback to use in the narrative.

**Agent F**: read `roi.json` for all pre-computed figures. Present and explain each benefit category with the actual computed numbers from the formulas in `06_roi.md`.

### Step 4 — Consistency review and refinement

Once all agents complete, the main agent reads all sections and checks:

1. **Consistency**: customer name, facility name, survey date, sensor count, gateway count, financial figures must match everywhere
2. **Repetition**: if two sections cover the same concept, keep it in the earlier section and shorten/remove the duplicate
3. **Completeness**: every output file exists and has substantive content
4. **Headers**: verify that multi-subsection files (03, 04, 06, 07) have the exact UPPERCASE divider lines the render script expects
5. **Placeholders**: collect all `[DA COMPILARE: ...]` to report to the user

Revise the affected files directly.

---

### Step 5 — Render the DOCX

```bash
uv run python <skill_dir>/scripts/render_docx.py \
    docs/reports/<slug>/tmp_data/ \
    docs/reports/<slug>/output/report.docx
```

The script reads metadata from `qa.json` and `sizing.json`, builds the pricing table, splits multi-section files by the uppercase dividers, and renders the template. It prints any unfilled `{{ }}` placeholders.

If a subsection comes out empty, check that the corresponding uppercase divider in the section file exactly matches what the script expects (see the `parse_section_*` functions in the script).

---

### Step 6 — Report to user

- Output file path
- Key metrics: sensor count, gateway count, ROI %, payback months
- Any `[DA COMPILARE]` placeholders requiring human input
- Any sections where data was sparse or defaults were used
